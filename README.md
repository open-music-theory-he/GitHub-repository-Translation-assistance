# GitHub Repository Translation Assistance

אוטומציה בת שני שלבים לתרגום קבצי ה-Markdown של המאגר
[`open-music-theory-he/open-music-theory-he.github.io`](https://github.com/open-music-theory-he/open-music-theory-he.github.io)
לעברית, באמצעות Gemini API. הכל רץ כ-GitHub Actions **במאגר הזה בלבד** –
המאגר המקורי נקרא ממנו לקריאה בלבד ולעולם לא נכתב אליו.

## מבנה

```
.github/workflows/
  extract-sentences.yml   # סקריפט 1 - הפעלה חד-פעמית ידנית
  translate.yml           # סקריפט 2 - רץ אוטומטית כל שעתיים
  dedup-terms.yml         # סקריפט 3 - הפעלה חד-פעמית ידנית
  push-translations.yml   # סקריפט 4 - הפעלה חד-פעמית ידנית (פותח PR)
scripts/
  extract_sentences.py
  translate_sentences.py
  remove_duplicate_terms.py
  push_translations.py
data/
  sentences.json              # פלט של סקריפט 1
  translations.json           # פלט מצטבר של סקריפט 2
  translations_deduped.json   # פלט של סקריפט 3
  dedup_log.txt               # יומן של סקריפט 3
requirements.txt
```

## הגדרה חד-פעמית (חובה לפני ההרצה)

1. **Secret**: Settings → Secrets and variables → Actions → New repository
   secret בשם `GEMINI_API_KEY` עם מפתח ה-API של Gemini שלך.
2. **הרשאות כתיבה**: Settings → Actions → General → Workflow permissions →
   בחר "Read and write permissions" ושמור. זה נדרש כדי שה-workflow-ים יוכלו
   ל-commit ול-push את קבצי ה-JSON בחזרה למאגר הזה (הם משתמשים ב-
   `GITHUB_TOKEN` המובנה, אין צורך ב-PAT נפרד).
3. מכיוון שמאגר המקור ציבורי, אין צורך בטוקן כדי לקרוא ממנו. אם בעתיד הוא
   יהפוך לפרטי, יש להוסיף secret בשם `SOURCE_REPO_TOKEN` (PAT עם הרשאת קריאה
   למאגר המקור) ולהסיר את ההערה בשורות המתאימות ב-`extract-sentences.yml`.

> **הערה לגבי האסימונים ששיתפת בשיחה**: כדי לא לשמור סודות בקוד, בחרתי שלא
> להטמיע את ה-PAT של GitHub או את מפתח ה-Gemini בקבצים עצמם – רק להשתמש ב-
> secrets כמתואר למעלה. מומלץ בכל מקרה **לסבב (rotate)** את שני האסימונים
> ששלחת בצ'אט, כדי שלא יישארו תקפים אם מישהו אחר יראה את ההיסטוריה.

## הרצה

1. הפעל ידנית את workflow **"1 - Extract sentences from source repo"** מתוך
   לשונית Actions (Run workflow). הוא יווצר את `data/sentences.json`.
2. workflow **"2 - Translate sentences (Gemini)"** ירוץ אוטומטית כל שעתיים
   (אפשר גם להריץ אותו ידנית לבדיקה). בכל הרצה:
   - אם `data/translations.json` מסומן `completed: true` – הסקריפט יוצא מיד
     ולא עושה כלום.
   - אחרת הוא ממשיך מהמשפט הראשון שעדיין אין לו תרגום.
   - יש השהיה קצרה בין קריאה לקריאה, ו-backoff גדל בין ניסיונות חוזרים על
     אותו משפט (ברירת מחדל: 60 שניות אחרי כישלון ראשון, 300 שניות אחרי כישלון
     שני; כישלון שלישי ברציפות = שמירה ועצירה עד ההרצה הבאה).
   - אחרי 30 דקות מתחילת ההרצה, הסקריפט מסיים את המשפט הנוכחי ועוצר, גם
     בהיעדר שגיאות.

**חשוב**: עצירה של הרצה לפני שכל הקבצים תורגמו (בגלל מגבלת ה-30 דקות, או
בגלל 3 כישלונות רצופים) היא **התנהגות תקנית וצפויה**, לא שגיאה - ה-workflow
תמיד יסתיים ב-exit code 0 (ירוק) במקרים האלה, וימשיך מאותה נקודה בהרצה
הבאה. שגיאה אמיתית (כמו `GEMINI_API_KEY` חסר לגמרי) עדיין תיכשל בצורה
ברורה, כי זו אכן בעיית הגדרה שכדאי לדעת עליה מיד.

### שינוי מודל / השהיה בקלות

הכי קל לשנות את המודל או את זמני ההשהיה **בלי לגעת בקוד או בקובץ ה-workflow
בכלל**: Settings → Secrets and variables → Actions → לשונית **Variables**
(לא Secrets) → הוסף repository variable עם אחד מהשמות הבאים:

| שם המשתנה | ברירת מחדל | משמעות |
|---|---|---|
| `GEMINI_MODEL` | `gemini-3.5-flash` | איזה מודל Gemini להשתמש בו |
| `SLEEP_BETWEEN_SECONDS` | `4` | השהיה בין קריאה מוצלחת לקריאה הבאה |
| `TIME_BUDGET_SECONDS` | `1800` | כמה זמן (בשניות) כל הרצה רצה לפני שהיא עוצרת |
| `RETRY_DELAYS_SECONDS` | `60,300` | זמני ההמתנה בין ניסיונות חוזרים (רשימה מופרדת בפסיקים) |

אם לא מגדירים variable כזה, נעשה שימוש בברירת המחדל אוטומטית - אין צורך
לגעת בכלום כדי שהכל ימשיך לעבוד כרגיל.

## שלב 3: הסרת מונחים כפולים באנגלית

הפעל ידנית את workflow **"3 - Remove duplicate English terms"** אחרי
שסקריפט 2 סיים (`completed: true`). הוא קורא את `data/translations.json`,
ולכל קובץ מקור בנפרד מוחק מופעים חוזרים של אותו ביטוי אנגלי בסוגריים
עגולים (למשל `(Sub-Dominant)`) - ומשאיר רק את המופע הראשון בכל קובץ.

תנאים למחיקה (כולם חייבים להתקיים):
- סוגריים עגולים בלבד.
- המופע חוזר לפחות פעמיים באותו קובץ.
- אין אותיות בעברית בתוך הסוגריים.
- לפחות 4 אותיות אנגליות בתוך הסוגריים (רווחים/פיסוק לא נספרים).
- לא קישור (גם אם התוכן עצמו נראה כמו URL, וגם אם זה חלק מקישור Markdown
  `[טקסט](url)`).

פלט (אף פעם לא דורס את `translations.json` המקורי):
- `data/translations_deduped.json` - עותק מלא עם התיקונים.
- `data/dedup_log.txt` - יומן קריא לבני אדם של כל מה שנמחק, לפי קובץ.

בנוסף, אותו סקריפט מבצע גם תיקון קטן ונפרד: כל מופע של "מייג'ור" (בכל
צורות הגרש/גרשיים הנפוצות - `'`, `׳`, `’`) הופך ל"מז'ור". גם זה מתועד
ב-`dedup_log.txt`, כשורה נוספת לכל קובץ שבו בוצע השינוי, למשל:
```
a.md
(Sub-Dominant)
 Deleted 1 times in this file
 Replaced "מייג'ור" with "מז'ור" 3 times in this file
```

ותיקון קטן נוסף: הברות סולפז' (Do/Re/Mi/Fa/Sol/So/La/Si/Ti) שנשארו באנגלית
בתוך התרגום מומרות לעברית לפי המיפוי הבא:

| אנגלית | עברית |
|---|---|
| Do | דו |
| Re | רה |
| Mi | מי |
| Fa | פה |
| Sol, So | סול |
| La | לה |
| Si, Ti | סי |

אבל **רק** אם:
- זו המילה כולה ולא חלק ממילה אחרת (למשל `Domain` או `Solo` לעולם לא ייפגעו).
- המילה הסמוכה (מתעלמים מרווחים/פיסוק) **אינה כבר** התרגום העברי - כלומר
  אם יש כבר גלוסה בעברית צמודה (לפני או אחרי), כמו `Do (דו)`, זה נשאר
  ללא שינוי כדי לא ליצור כפילות.

דוגמה:
```
התו Do הוא התו הראשון          ->  התו דו הוא התו הראשון
התו Do )דו) הוא התו הראשון      ->  (ללא שינוי - הגלוסה כבר שם)
```
גם זה מתועד ב-`dedup_log.txt`.


## שלב 4: דחיפת התרגום למאגר המקור (דרך Pull Request)

הפעל ידנית את workflow **"4 - Push translations to source repo (via Pull
Request)"** אחרי שסקריפט 3 סיים. הוא קורא את `data/translations_deduped.json`,
ולכל קובץ מקור עובר על יחידות התרגום לפי סדר ה-id: מחפש את `source_sentence`
בטקסט הקובץ (העותק המקומי שנמשך מהמאגר), ומחליף רק את המופע **הראשון**
שנמצא ב-`translated_sentence`, וממשיך ליחידה הבאה.

**חשוב: הסקריפט הזה לעולם לא דוחף ישירות ל-branch הראשי של מאגר המקור.**
בהתאם למה שסיכמנו, ה-workflow פותח **Pull Request** במאגר המקור עם כל
ההחלפות (על גבי branch בשם `hebrew-translation-import`), כדי שתוכל לעבור על
השינויים ולאשר אותם לפני שהם נכנסים בפועל. אם משפט מקור לא נמצא בקובץ
(למשל כי הקובץ השתנה מאז שסקריפט 1 רץ), הוא מדלג עליו עם אזהרה ברורה בלוג -
שווה לבדוק את זה לפני מיזוג ה-PR.

### הגדרה נוספת הנדרשת לשלב הזה

צריך secret נוסף בשם **`SOURCE_REPO_TOKEN`** (בנוסף ל-`GEMINI_API_KEY`
הקיים) - Personal Access Token שיש לו הרשאות **כתיבה** למאגר המקור
`open-music-theory-he/open-music-theory-he.github.io`:
- אם זה fine-grained PAT: הרשאות `Contents: Read and write` ו-
  `Pull requests: Read and write` על המאגר הזה בלבד.
- אם זה classic PAT: scope `repo`.

ה-secret הזה שונה מהטוקן ששלחת בהתחלה (שלא נעשה בו שימוש בכלל בפרויקט
הזה) - יש ליצור טוקן חדש עם ההרשאות המתאימות ולהוסיף אותו כ-repository
secret בשם `SOURCE_REPO_TOKEN` במאגר היעד (זה שבו נמצאים ה-workflows).

`gemini-3.1-pro-preview` (לאיכות מקסימלית) בקובץ ה-workflow.

## הנחות/מגבלות בסקריפט 1

- "משפט" מוגדר כאן כשורה לא-ריקה בקובץ ה-MD (בהתאם לניסוח "יעבור שורה
  אחר שורה" בתיאור סקריפט 2).
- מוסר: כותרות (`#`...), סימון ציטוט (`>`), תבליט רשימה (`-`/`*`/`+`/`1.`
  בתחילת שורה בלבד, פעם אחת), קווי הפרדה אופקיים (`---`/`***`/`===`), שורות
  הפרדה של טבלאות (`| --- | --- |`), YAML front-matter של Jekyll, ובלוקי קוד
  מגודרים (` ``` `).
- נשמר ללא שינוי: כל עיצוב שאינו ברמת השורה כולה - `**bold**`, `*italic*`,
  ` `code` `, קישורים `[text](url)` וכו', גם אם במקרה מסוים הם משתרעים על פני
  השורה כולה.
- שורות תוכן של טבלאות (לא שורות ההפרדה) נשמרות כמות שהן, כולל תווי `|`.
- **שורות ללא מילים אמיתיות לתרגום** (למשל `====`, מספרים בלבד, או תגי
  Liquid/Jekyll עצמאיים כמו `{{ page.title }}`) **מסוננות כברירת מחדל**
  ולא נכתבות ל-`sentences.json` כלל, כדי לא לבזבז קריאות API על שורות שאין
  בהן בכלל טקסט לתרגם. אפשר לבטל את הסינון הזה עם `--include-no-word-lines`
  (או ה-checkbox המתאים כשמריצים את ה-workflow ידנית). הבדיקה משאירה מילים
  אמיתיות על כנן גם כשהן משולבות עם קוד - למשל `Welcome, {{ user.name }}!`
  עדיין ייחשב כמשפט תקין ויישמר (כי יש בו את המילה "Welcome").

אם תרצה התנהגות שונה (לדוגמה: פיצול לפי משפטים דקדוקיים אמיתיים ולא לפי
שורות, או טיפול שונה בטבלאות) – אפשר לעדכן את `scripts/extract_sentences.py`
בהתאם.

## שורות קוד/לא-פרוזה בסקריפט 2

הפרומפט לתרגום כולל כלל נוסף (מס' 9): אם השורה כולה היא קוד, ביטוי תבנית
(כמו Liquid), פקודת מעטפת, כתובת URL גולמית וכו', ו**אין בה שום טקסט אנושי
לתרגום כלל**, המודל מתבקש להחזיר טוקן קבוע (`NO_TRANSLATION_NEEDED`) במקום
לנסות "לתרגם" את זה. הסקריפט מזהה את הטוקן הזה, שומר את הטקסט המקורי כפי
שהוא כ"תרגום", ומדפיס הודעה ברורה בטרמינל:

```
[id=X] file=...
  >>> MODEL FLAGGED THIS AS CODE/NON-PROSE - KEEPING ORIGINAL TEXT AS-IS <<<
  INPUT/OUTPUT (unchanged): ...
```

כדי לא לפגוע בתרגומים אמיתיים בטעות: הזיהוי דורש שהתשובה תהיה **אך ורק**
הטוקן הזה (עם אפשרות למרכאות/גרשיים מסביב), ולא רק שהטוקן מופיע איפשהו בתוך
תשובה ארוכה יותר - כך שאם המודל בכל זאת מחזיר תרגום אמיתי, הוא תמיד יישמר
כמות שהוא.
